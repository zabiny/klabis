📋 Member Self-Edit & Admin Edit API - Proposal Summary

GitHub Issue: https://github.com/zabiny/klabis/issues/4

Change ID: add-member-self-edit-api

Status: Ready for Review

Branch: specs

🎯 Executive Summary

Problem: Members currently cannot update their own personal information. All updates require administrator intervention, creating overhead and potential delays.


Solution: Implement a member self-edit API that allows members to update their own information, with administrators able to edit core data (name, date of birth) when needed.


Impact: Reduces administrative workload, improves data accuracy, and enhances member satisfaction while maintaining data security through role-based access control.

1. What's Changing

🆕 New API Endpoint

PATCH /api/members/{id} - Update member information

👤 Two Authorization Models

Member Self-Edit

  • Members can only edit their own information
  • Access matched via OAuth2 subject
  • Limited set of editable fields
  • Prevents accidental data corruption

Admin Edit

  • Users with MEMBERS:UPDATE permission
  • Can edit any member's information
  • Additional fields: firstName, lastName, dateOfBirth
  • For data corrections and critical updates

2. Important Validation Rules

✅ Contact Requirements

  • At least one email must be provided (either member's own OR guardian's)
  • At least one phone must be provided (either member's own OR guardian's)
  • This ensures we can always reach the member or their legal guardian

⚠️ Document Expiry Validation

Important: Expiry date validation only applies when setting/updating a document.

  • Member CAN update phone number even if their ID card is expired
  • Member CAN update address even if their trainer license is expired
  • Validation prevents setting expired documents, but doesn't block other updates
  • This provides better user experience while maintaining data integrity

🇨🇿 Nationality-Dependent Fields

  • Rodné číslo (Czech ID number) field is only available when Czech nationality is selected
  • If nationality changes from Czech to non-Czech, the rodné číslo field is cleared
  • This is a conditional field based on nationality selection

3. Technical Decisions

🔧 Decision 1: PATCH vs PUT

Chosen: PATCH /api/members/{id} (partial updates)

✅ Why PATCH (Benefits)

  • Bandwidth efficient: Only send changed fields
  • Better UX: Matches real-world editing (1-3 fields typically)
  • Mobile-friendly: Less data transfer
  • Prevents data loss: Concurrent edits to different fields don't overwrite each other
  • Industry standard: Modern RESTful API best practice

❌ Trade-offs

  • Higher complexity: ~4-5x more code than PUT (~40-50 LOC vs ~10 LOC)
  • Merge logic: Must carefully merge partial data with existing state
  • Null semantics: Must define what null means (we chose: null = ignore field)

Exit Strategy: Can add PUT endpoint later if needed (low complexity). PATCH can be deprecated if it proves overkill.

🏗️ Decision 2: Value Objects for Complex Data

Chosen: Domain-Driven Design (DDD) value objects instead of scattered primitive fields

Value Object Contains Benefits
IdentityCard cardNumber + validityDate Encapsulation, validation at construction, immutable
MedicalCourse completionDate + optional validityDate Keeps related data together, ensures validity after completion
TrainerLicense licenseNumber + validityDate Type-safe, reusable, self-validating
Address street + city + postalCode + country Already exists, proven pattern

Benefits: Better encapsulation, type safety, validation enforcement, easier testing, self-documenting code.

4. Implementation Overview

📊 What Will Be Built

  1. Domain Model
    • 3 new value objects: IdentityCard, MedicalCourse, TrainerLicense
    • 1 new enum: DrivingLicenseGroup
    • 1 new text field: dietaryRestrictions
  2. API Endpoint
    • PATCH /api/members/{id} with role-based field access
    • HATEOAS hypermedia controls (HAL+FORMS format)
    • Dynamic templates based on user role (member vs admin)
  3. Authorization
    • Self-edit: OAuth2 subject must match member ID
    • Admin edit: MEMBERS:UPDATE permission required
    • Role-based field filtering (members can't edit admin-only fields)
  4. Database
    • New columns for each value object
    • Optimistic locking (version column) to prevent concurrent update conflicts
    • Indexes on expiry date fields for future queries

🔄 API Response Format

Example Request:

PATCH /api/members/123 
{
  "phone": "+420 987 654 321",
  "drivingLicenseGroup": "B"

  "_links": {

    "collection": {"href": "/api/members"}
  }
}

5. Impact & Benefits

✅ Business Benefits

  • Reduced admin workload: Members update their own data
  • Faster updates: No waiting for admin intervention
  • Better data accuracy: Members keep their info current
  • Improved member satisfaction: Self-service capability
  • Dietary requests handled: Food accommodation planning

🔒 Security & Control

  • Role-based access: Members can only edit their own data
  • Admin oversight: Core fields (name, DOB) remain admin-only
  • Audit trail: All changes logged with user context
  • Optimistic locking: Prevents concurrent update conflicts
  • Field filtering: Non-admins can't access admin-only fields

6. Risks & Mitigations

Risk Mitigation
Members accidentally break their data Strict validation rules, admin can still edit core fields (name, DOB), optimistic locking prevents concurrent overwrites
Higher implementation complexity Comprehensive testing, clear documentation, value objects encapsulate validation logic, can fall back to PUT if needed
Expired documents in database Accepted design choice - allows updating other fields, UI can show warnings, background job can flag expired documents
Mobile app performance PATCH reduces bandwidth (only send changed fields), value objects keep payloads small
Concurrent edits causing conflicts PATCH only updates specified fields (other fields untouched), optimistic locking (version column) detects conflicts

7. Success Criteria

We will validate the success of this implementation based on:

  • ✅ Usage metrics: Average fields updated per request < 5 (indicates PATCH is appropriate)
  • ✅ Data integrity: Race condition data loss issues < 1 per month
  • ✅ User feedback: Mobile app and web performance acceptable
  • ✅ Team velocity: Complexity doesn't slow future development
  • ✅ Admin workload: Measurable reduction in manual update requests
