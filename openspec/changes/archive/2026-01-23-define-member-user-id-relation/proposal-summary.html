📋 Define Member-User ID Relationship - Proposal Summary

Change ID: define-member-user-id-relation

Status: Ready for Stakeholder Review

Type: Design Clarification / Architectural Enhancement Internal Only

Proposed: 2025-01-18

🎯 Executive Summary

Problem: Currently, there's ambiguity in how member and user entities relate to each other regarding their IDs. The system needs a clear design pattern for the Member-User relationship.


Solution: Establish that every Member has an associated User, and the User's ID is used as the Member's ID. Introduce a UserId Java record as a type-safe identifier shared between both aggregates.


Impact: Simplifies relationship management, improves query performance, and establishes a clear DDD pattern for future aggregate identification.


Important: This is an internal implementation change only. No API changes - clients continue to use the same member registration endpoint. The User is created automatically as part of the member registration flow.

1. What's Changing

🔗 Core Design Rule

Rule: Member.id == User.id for all members

Every Member entity is created with an associated User entity, and both share the same UserId instance.

🆕 New Component: UserId Value Object

UserId Java Record

  • Location: users.domain package
  • Type: Java record (immutable, type-safe) Record
  • Purpose: Prevents accidental mixing of IDs from different aggregates
  • Used by: Both User and Member aggregates

📊 Member Registration Flow Changes

Aspect Current State Proposed State
ID Generation Member gets generated UUID independently Member uses User's UserId (shared UUID)
User Creation Manual or separate process Automatic as part of member registration
API Endpoint POST /api/members No change - same endpoint
API Parameters Member data only No change - same parameters
Implementation Direct member creation User created first, then Member with shared UserId
Query Performance Requires joins between tables Direct lookup by shared ID (no joins)

⚠️ No API Changes

Important: Internal Implementation Only

This proposal does NOT change the API contract.

  • Endpoint: Still POST /api/members
  • Request body: Still contains member data only (no user_id parameter)
  • Response: Same format with HAL+FORMS links
  • Change: Only internal - RegisterMemberCommandHandler now creates User first, then Member

Client Impact: None. Clients don't need to change anything.

2. Architecture & Design Decisions

🏗️ Decision 1: Shared ID Pattern (Member ID = User ID)

Chosen: Member and User share the same UserId instance

✅ Benefits

  • Simplified queries: No JOIN operations needed to get member by user ID
  • Clear relationship: 1:1 mapping is explicit and enforced
  • Performance: Direct primary key lookups
  • Consistency: Cannot have orphaned members or users without clear association
  • DDD pattern: Aggregate root identity is clear and consistent

❌ Trade-offs

  • Module dependency: Members module depends on Users module (for UserId)
  • Tighter coupling: Member lifecycle tied to User lifecycle
  • Creation order: User must be created before Member

Mitigation: Module dependency is natural (members are created for users). UserId is a stable, immutable value object safe to share across modules.

🔧 Decision 2: UserId as Java Record

Chosen: Java record instead of class or plain UUID

✅ Why Java Record

  • Type safety: Compiler prevents mixing UserId with other ID types
  • Immutability: Records are immutable by design
  • Concise: 1 line of code vs 15+ for equivalent class
  • Auto-generated: equals(), hashCode(), constructor, toString()
  • Domain clarity: Explicitly expresses this ID belongs to User domain
  • Future-proof: Easy to extend when other aggregates are added

❌ Trade-offs

  • Additional type: One more class to maintain
  • Module dependency: Members module imports from Users module

Exit Strategy: Can be refactored to separate IDs if the 1:1 relationship changes. However, this is unlikely given the domain model.

📦 Decision 3: UserId Location in users.domain Package

Chosen: com.klabis.users.domain.UserId

✅ Benefits

  • DDD principle: UserId belongs to Users bounded context (primary domain)
  • Clear ownership: Users module owns this core type
  • Logical dependency: Members depending on Users is natural

❌ Trade-offs

  • Cross-module import: Members module imports from Users module

Alternative Considered: shared.domain package - Rejected because UserId has primary meaning in Users domain, not a generic shared concept.

3. Technical Implementation

📝 UserId Record Structure

package com.klabis.users.domain;

public record UserId(UUID uuid) {
    public UserId {
        if (uuid == null) {
            throw new IllegalArgumentException("UUID cannot be null");
        }
    }

    public static UserId fromString(String uuidString) {
        try {
            return new UserId(UUID.fromString(uuidString));
        } catch (IllegalArgumentException e) {
            throw new IllegalArgumentException("Invalid UUID string: " + uuidString);
        }
    }
}

                     │

                     │
                     ▼
┌─────────────────────────────────────────────────────────┐

└─────────────────────────────────────────────────────────┘

Key Points:

  • User created first with generated UserId
  • Member created second using the same UserId
  • Both operations in single transaction
  • If User creation fails, Member creation is not attempted

📊 What Will Be Built

  1. UserId Value Object
    • Java record in users.domain package
    • Compact constructor with null validation
    • Static factory method: fromString(String uuidString)
    • Unit tests for validation, equals/hashCode
  2. Domain Model Updates
    • User aggregate uses UserId instead of UUID
    • Member aggregate uses UserId instead of UUID

  4. Database
    • Constraints to establish shared ID relationship
    • Optional: Foreign key constraint
  5. Repository & Query Optimization
    • Leverage shared ID for direct lookups
    • No JOIN operations for member-by-user queries

6. Risks & Mitigations

Risk Mitigation
Module coupling
Members depending on Users		 Natural and acceptable dependency. Members are created for Users, so the dependency reflects the domain reality. UserId is stable and immutable.
User creation failure blocking member creation Acceptable - if User cannot be created, Member should not exist. Transactional rollback ensures data consistency.
Performance impact of User-first creation Negligible. User creation is lightweight (UUID generation, minimal fields). Same transaction context minimizes overhead.
Complexity increase Minimal. Records are simple, and the pattern is well-established. Developer onboarding is straightforward.
Testing overhead Low. UserId is simple to test. Additional test for User-then-Member creation flow is straightforward.
API compatibility No risk - No API changes. Clients continue working without modifications.
