1. Cíl studie a metodika

Cílem je posoudit, zda lze REST API vrstvy modulů members a calendar nahradit Spring Data REST (dále SDR). Studie:

  • Zmapovala každý endpoint, HATEOAS processor, Jackson customizaci a perzistenční pattern v obou modulech.
  • Porovnala potřeby s aktuálními schopnostmi SDR (dokumentace Spring Data REST, main branch, ověřeno přes Context7).
  • Klasifikovala jednotlivé funkce podle obtížnosti migrace: LEHKÉ STŘEDNÍ VELMI TĚŽKÉ BLOKER

2. Celkový závěr (TL;DR)

Migraci nelze provést přímočaře. SDR je silný nástroj pro CRUD expozici agregátů, ale Klabis staví REST vrstvu na řadě konceptů, které SDR nativně neposkytuje a které tvoří podstatu produktu (field-level autorizace, domain-aware HAL-FORMS afordance, custom business operace, PATCH sémantika přes PatchField<T>, polymorfní agregáty, cross-module koordinace přes eventy).

Reálný scénář migrace je hybrid: u jednoduchých čtení nahradit vlastní controller expozicí repository přes SDR, u všeho ostatního si ponechat vlastní @RestController. Přínos takového hybridu je však marginální — úspora ~5 % kódu za cenu nekonzistence stylu, duplicitní konfigurace a limitů SDR, které při dalším vývoji narážejí na strop. Doporučení: migraci neprovádět.

4. Kompatibilita Spring Data REST × Spring Data JDBC

Formálně podporováno, prakticky okrajové. Spring Data REST oficiálně podporuje "Spring Data backend" (JPA, MongoDB, JDBC, Neo4j, Cassandra, Gemfire, Solr). Reálné nasazení s Spring Data JDBC je však minoritní a dokumentace se opírá o JPA příklady.

Konkrétní důsledky pro Klabis:

  • Memento pattern. SDR by exponoval MemberMemento / CalendarMemento — tedy persistenční objekty místo doménových agregátů. To přímo narušuje hexagonální architekturu a DDD vrstvení, které je základem projektu.
  • Rekonstrukce agregátu. SDR volá CrudRepository.save() přímo; neprochází přes doménové metody jako member.update(cmd), kde se vyhodnocují invarianty a registrují domain eventy. Invarianty by tím skončily až na úrovni databáze.
  • Polymorfismus. CalendarMemento má kind discriminator s převodem na ManualCalendarItem / EventCalendarItem. SDR pracuje s jedním typem, tudíž by modul musel být rozdělen na dva resource nebo přejít na plochý model.
  • ID jako value object. MemberId, CalendarItemId, UserId jsou records obalující UUID. SDR vyžaduje @Id na poli repozitované entity — fungovat bude, ale výstup self linku bude nad raw UUID, ne doménovým ID.
Klíčový architektonický konflikt: Klabis záměrně odděluje doménu od perzistence pomocí Memento + Adapter. SDR toto oddělení ruší a vrací vás k anemickému modelu "JPA entita = REST resource". To je krok zpět od DDD stylu, který projekt praktikuje.

GET detail LEHKÉ Nativní.

6. Modul members — analýza & návrh

Klíčové charakteristiky:

  • 6 endpointů, z toho 3 business akce (/suspend, /resume, registrace).
  • PATCH se sémantikou PatchField<T> + per-field @HasAuthority (14 polí).
  • Field-level autorizace v response DTO (@OwnerVisible, NullDeniedHandler).
  • Cross-module závislost: RegistrationService vytváří User + Member v jedné transakci.
  • Veto event pattern na /suspend.
  • GDPR audit eventy (BirthNumberAccessedEvent) publikované z getMember().
Verdikt: migraci nedoporučit. Celkem 6 funkcí klasifikovaných jako BLOKER nebo VELMI TĚŽKÉ, 1 LEHKÁ (GET list). Migrací by vzniklo ~5 % úspora kódu a přibyla by 2. REST technologie. Čistší postup je refaktoring stávajícího controlleru, případně extrakce helperů do common/ui.

Mapování jednotlivých endpointů

EndpointSDR?Poznámka
GET /api/members⚠️SDR zvládne paging. Field-level filtrování pro non-admin vyžaduje vlastní processor (= stejný kód).
GET /api/members/options✅Projekce + search endpoint.
GET /api/members/{id}❌GDPR audit event při čtení rodného čísla nelze publikovat bez mezivrstvy.
PATCH /api/members/{id}❌PatchField + field-level auth. SDR nemá mechanismus.
POST /api/members/{id}/suspend❌Veto event, cross-module (UserService), není CRUD.
POST /api/members/{id}/resume❌Cross-module business akce.
POST /api/members (registrace)❌Multi-aggregate transakce (User + Member).

7. Modul calendar — analýza & návrh

Klíčové charakteristiky:

  • 5 endpointů — klasické CRUD.
  • Polymorfní agregát s discriminatorem kind.
  • EventCalendarItem je read-only, řízeno assertCanBeDeleted().
  • Filtrování date-range s validací ≤366 dní + month-based prev/next navigační linky.
  • Reconciliace z event eventů (EventPublishedEvent, …).
  • HAL-FORMS afordance dle subtypu: Manual = UPDATE + DELETE; Event = žádné.
Verdikt: částečně proveditelné, ale bez přínosu. SDR zvládne ~60 % CRUD, ale polymorfní handling, month navigace a read-only enforcement stejně vyžadují custom processor + event handler. Úspora kódu ≈ 15 %, výměnou za nutnost re-fetch z DB v @HandleBeforeSave/@HandleBeforeDelete a rozdělení jednoho endpointu na dva resource.

Mapování endpointů

EndpointSDR?Poznámka
GET /api/calendar-items⚠️Date-range search možný, ale validace rozsahu a prev/next linky = custom kód.
GET /api/calendar-items/{id}⚠️Polymorfní afordance vyžadují custom processor + re-fetch z DB.
POST /api/calendar-items✅Normalizace přes @HandleBeforeCreate.
PUT /api/calendar-items/{id}⚠️Read-only check přes @HandleBeforeSave s re-fetch.
DELETE /api/calendar-items/{id}⚠️Jako PUT.

9. Hypotetický postup hybridní migrace

Pokud by se přesto přistoupilo k částečné migraci (čistě teoreticky):

  1. Fáze 1 — Pilot. Vybrat nejjednodušší modul (ne members, ne calendar). Kandidáti: audit log readonly, reference data.
  2. Fáze 2 — Calendar GET. Ponechat POST/PUT/DELETE v custom controlleru, u GET zkusit SDR + projekci. Porovnat výstup HAL-FORMS proti dnešku.
  3. Fáze 3 — Rozhodnutí. Pokud vyjde ≥30 % úspora bez ztráty funkcionality, jít dál. Jinak migraci zastavit.
  4. Fáze 4 (nedoporučeno) — Members. Pouze /options endpoint jako search resource. Všechny ostatní endpointy ponechat.

Předpokládaný odhad úsilí pilotní fáze: 3–5 člověkodnů. Očekávaný přínos: validace/vyvrácení hypotézy.

10. Doporučení

Migraci modulů members a calendar na Spring Data REST nedoporučujeme.

Proč:

  1. Architektonický mismatch. SDR exponuje persistenční objekty (memento); Klabis staví na oddělení domény od persistence. Migrace by znamenala krok zpět k anemickému modelu.
  2. Čistý přínos blízký nule. CRUD tvoří v members modulu cca 1 endpoint z 6; v calendaru 5 z 5, ale všechny 5 mají business rules vyžadující stejný custom kód jako dnes.
  3. SDR nepokrývá klíčové features. Field-level autorizace, PATCH sémantika, veto events, polymorfní agregáty, domain-aware afordance — pro všechny by bylo nutné napsat vlastní kód nad SDR.
  4. Dvojí REST stack. Hybrid = @RestController i @RepositoryRestController vedle sebe, dvě OpenAPI konfigurace, dvě testovací strategie.
  5. Riziko stagnace SDR. Spring Data REST je v posledních releasech ve stabilizačním režimu; HAL-FORMS customizace nesledují rychlý vývoj Spring HATEOAS.

Alternativa — refactoring stávajícího řešení:

  • Extrahovat do common/ui opakující se kódy paging/sort validace (dnes v MemberController i CalendarController).
  • Standardizovat pattern "ModelWithDomainPostprocessor + afordance dle stavu" — např. fluent builder AffordancesBuilder.
  • Podchytit month-based navigaci v calendaru jako helper.
  • Review pokrytí OpenAPI dokumentace (použití @SecurityRequirement).
