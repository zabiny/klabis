Analýza migrace Klabis frontendu z React na HTMX

Datum: 2026-04-19 · Autor: analýza Claude Code · Projekt: klabis

TL;DR. HTMX je technicky reálná alternativa k současnému React SPA: ve Spring Boot existuje oficiálně udržovaná knihovna htmx-spring-boot (wimdeblauwe) poskytující @HxRequest, HtmxResponse, HxTrigger, OOB swapy a fragment rendering přes Thymeleaf. Naše API zůstává zachováno (HAL+FORMS je výborný kontrakt pro 3rd party). Migrace ale není „přelepení šablon“ — klíčové překážky jsou OAuth2 authorization-code flow s PKCE (access token v cookie místo sessionStorage), HAL+FORMS → Thymeleaf template přepis, formulářová UX (Formik + Yup → server-side validace + HTML5), klientské interakce (admin mód, theme toggle, modal state). Doporučený postup: zavést paralelní Thymeleaf/HTMX modul vedle React SPA, migrovat po stránkách a sdílet layout přes hx-boost.

1. Kontext a cíle

  • Současný stav: Spring Boot 4.0.5 backend s HAL+FORMS API + samostatný React 19 SPA (Vite, TanStack Query, Formik, Tailwind, oidc-client-ts).
  • Motivace pro HTMX: snížení komplexity frontendu, eliminace duplicity typů (OpenAPI generator), rychlejší vývoj funkcionalit, méně klientského stavu, odstranění buildu a dev serveru.
  • Podmínka: HAL+FORMS API se zachovává pro 3rd party integrace (ORIS, externí klienti).

2. Podpora HTMX ve Spring Boot

2.1 Oficiální knihovna htmx-spring-boot (Wim Deblauwe)

Aktivně udržovaná knihovna (GitHub: wimdeblauwe/htmx-spring-boot) poskytuje first-class integraci HTMX s Spring MVC + Thymeleaf.

FeaturePopis
@HxRequestAnotace pro mapování, která se aplikuje pouze na HTMX requesty (HX-Request: true). Umožňuje mít na stejné URL dvě metody — jednu pro plný render, druhou pro HTMX fragment.
HtmxRequest / HtmxResponseArgument-resolver objekty pro čtení HTMX hlaviček (target, trigger, current-url) a pro nastavení response hlaviček (HX-Trigger, HX-Retarget, HX-Reswap, HX-Push-Url, HX-Redirect, HX-Refresh, HX-Location).
Metodové anotace@HxTrigger, @HxPushUrl, @HxRedirect, @HxRefresh, @HxReselect, @HxReswap, @HxRetarget, @HxLocation — deklarativní nastavení response hlaviček bez injektování HtmxResponse.
View prefixyredirect:htmx:/path, redirect:htmx:location:/path, refresh:htmx — klientský redirect místo serverového.
OOB swapyFragmentsRendering.with("users :: list").fragment("users :: count").build() — jeden request vrátí více fragmentů, každý swapnutý do svého targetu (skvělé pro aktualizaci počítadla + seznamu současně).
Exception handlery@ExceptionHandler může přijmout HtmxRequest/HtmxResponse a rozhodnout, jestli chybu swapnout do #error-message nebo redirectnout.

2.2 Ostatní stavební bloky

  • Thymeleaf jako template engine (nativní podpora fragmentů přes th:fragment a th:replace), alternativně JTE nebo Mustache.
  • Spring Security — formulářové login + session cookie (HttpOnly, Secure, SameSite=Lax) nahrazuje OAuth2 bearer tokeny pro UI.
  • CSRF — obnoven, formulářový token vkládán do layoutu přes <meta name="_csrf">, HTMX ho pošle přes hx-headers.
  • Live reload — Spring Boot DevTools + Thymeleaf spring.thymeleaf.cache=false; žádný Vite není potřeba.
  • Hypermedia rozšíření: knihovny jako Alpine.js (10kB) nebo _hyperscript pokryjí klientské interakce (dropdown, theme toggle) bez React.

3. Přehled vlastností současného frontendu

Inventura toho, co aktuálně SPA dělá. Seznam vychází z frontend/src/App.tsx, HalNavigator2, KlabisTable, UI a pages.

3.1 Stránky (routes)

  1. Dashboard (HomePage) — role-based (AdminDashboard vs UserDashboard).
  2. Login (LoginPage) — vlastní formulář, posílá na /login Spring AS.
  3. Password setup / expired (2 stránky).
  4. Members — list, detail, registrace, finance účet, field-level edit (PatchField).
  5. Events — list, detail, category presets.
  6. Groups / Training Groups / Family Groups — list + detail, přidání členů/vlastníků/rodičů.
  7. Calendar — list kalendářních položek.
  8. Finances.
  9. GenericHalPage — catch-all navigator pro admin mode a neimplementované routes.

3.2 Sdílené komponenty

  • Layout — sidebar menu řízený HAL linky z root resourcu (useRootNavigation), theme toggle, admin toggle, logout.
  • KlabisTable + KlabisTableWithQuery — tabulka/karty s HAL embedded daty, řazení, pagination, card-view na mobilu.
  • HalNavigator2:
    • HalFormButton — tlačítko spouštějící formulář (modal nebo inline dle URL).
    • HalFormDisplay — render HAL+FORMS template, customLayout pro inline editing.
    • HalFormsForm / HalFormsFieldFactory — generují form fields z HAL+FORMS properties.
    • HalFormsPageLayout — orchestruje modal vs. URL-based form state.
    • MultiStepFormModal — wizardy (např. registrace člena).
    • HalEmbeddedTable, HalLinksSection, HalFormsSection.
  • UI/ — Button, Modal, Alert, Badge, Toast, Skeleton, Spinner, DetailRow, Card, ErrorDisplay, TokenRenewalOverlay.
  • Formuláře — Formik + Yup, klientská validace, BirthNumberConditionalField, SuspensionWarningDialog.

3.3 Klientské chování

  • OAuth2 Authorization Code + PKCE, access token v sessionStorage, silent renew přes refresh token.
  • TanStack Query cache + refetch on mutation.
  • React Router — client-side navigace, URL parametry pro inline formy (?form=xyz).
  • Admin mode toggle (Context API) — přepíná route handling.
  • Theme (light/dark) toggle (Tailwind).
  • Client-side validace (Yup schémata).
  • React localization (labels.ts).
  • Error boundary → ErrorFallback.

4. Mapa vlastností → HTMX ekvivalent

Každý řádek ukazuje, jak konkrétní SPA pattern migrovat.

Feature (React/SPA) HTMX řešení Náročnost
Client-side routing (React Router) hx-boost="true" na <body> — všechny <a>/<form> se mění na HTMX requesty, které swapnou <body> bez full reload. URL se aktualizuje přes hx-push-url. Back/forward button funguje nativně. Nízká
Layout + shared nav (Layout.tsx) Thymeleaf layout fragment (layout/main.html) + th:replace="~{::content}". Menu generováno serverově z HAL root resource (stejná logika jako dnes, jen na backendu). Nízká
KlabisTable (list + pagination + sort) Thymeleaf fragment table :: rows + server-side pagination. Sort/filter tlačítka posílají hx-get="/members?sort=lastName", hx-target="#members-table". Pagination přes stránky se swapem innerHTML. Nízká
Card-view na mobilu CSS media queries + dvě Thymeleaf fragments (table, cards) rendered současně, zobrazené responzivně. Žádná JS logika. Nízká
HalFormDisplay (generický render HAL+FORMS template) Neekvivalent 1:1. HTMX pracuje se server-rendered HTML formuláři — HAL+FORMS templaty se přepíšou na Thymeleaf šablony (nebo se vytvoří server-side generátor, který z HAL+FORMS properties vyrobí HTML form). Jednodušší varianta: pro UI se HAL+FORMS nepoužívá, jen pro 3rd party API. Vysoká
HalFormButton (otevření modal formuláře) <button hx-get="/members/new" hx-target="#modal" hx-swap="innerHTML"> + modal div na konci layoutu. Zavření modalu přes HX-Trigger: closeModal, který chytne Alpine.js / hyperscript. Střední
MultiStepFormModal (wizardy) Každý krok = vlastní endpoint, který vrací HTML fragment s formem pro další krok. Stav se drží v HTTP session (nebo v hidden fields). HTMX swapne celý wizard div. Střední
Inline editing (enrichTemplateWithReadOnlyFields + HalFormDisplay) Read-only view má tlačítko „Upravit“ → hx-get swapne řádek za editační form → submit swapne zpět read-only. Klasický HTMX vzor (click-to-edit example). Nízká
Formik + Yup (klient. validace) Server-side validace přes @Valid + Bean Validation. Inline chyby vrácené přes fragment: return "user/form"; při bindingResult.hasErrors(). HTML5 constraints (required, pattern, type="email") pro instant feedback. Volitelně hx-trigger="blur changed delay:300ms" pro validaci per-field. Střední
Conditional field (BirthNumberConditionalField) hx-get="/members/new/fragment/birth-number" hx-trigger="change from:#country" hx-target="#birth-number-wrapper" — server rozhodne, co renderovat. Nízká
TanStack Query cache + invalidation Odpadá. Mutace vrátí HX-Trigger: members-updated, cílový fragment poslouchá: hx-get="/members" hx-trigger="members-updated from:body". Nebo jednodušeji: OOB swap vrátí aktualizovaný seznam přímo v response mutace. Nízká
Toast notifications Server pošle HX-Trigger: {"toast": {"message": "Uloženo", "level": "success"}}. Global JS listener (Alpine/hyperscript) přidá toast do DOM. Nebo OOB swap do #toasts. Nízká
Admin mode toggle (Context API) Cookie/session atribut. Toggle volá hx-post="/user/preferences/admin-mode", backend přehodí flag a pošle HX-Refresh: true. Route handling už je na serveru. Nízká
Theme toggle (dark/light) Čistě klient: malý Alpine.js (~10kB) nebo vanilla JS snippet, který toggluje class="dark" na <html> + persistuje do localStorage. Tailwind dark mode funguje beze změny. Nízká
Error boundary (ErrorFallback) @ExceptionHandler vrátí error template. Pro HTMX requesty HxResponse.setRetarget("#error-zone"). Pro plné requesty whitelabel stránka. Nízká
Loading indicators (Skeleton, Spinner) HTMX má htmx-indicator class a hx-indicator=".spinner". Spinner se zobrazí jen během requestu. Nízká
Modal overlay (generický) Thymeleaf fragment modal.html swapovaný do #modal na <body>. Zavírání: hx-on::click="if (event.target === this) htmx.ajax('GET', '/empty', '#modal')" nebo hyperscript _="on closeModal remove me". Střední
Calendar page Server renderuje mřížku, navigace (prev/next month) přes hx-get="/calendar?month=...". Pro drag-and-drop události bude potřeba klientská knihovna (FullCalendar JS) — HTMX není pro tohle ideální. Střední
OAuth2/OIDC (oidc-client-ts) Hlavní architektonická změna. UI přestane volat API bearer tokenem a stane se resource server sám o sobě (session cookie). Viz sekce 6. Vysoká
GenericHalPage (catch-all navigator) Buď odpadá (každá stránka má vlastní server-rendered template), nebo se nahradí minimalistickým HTML navigatorem pro admin debug mode. Nízká
Localization (labels.ts) Přesun do Spring messages.properties + Thymeleaf #{member.firstName}. Jazyk už je jen čeština, takže migrace je přímočará. Nízká
Infinite scroll / lazy load (pokud bude potřeba) Nativní HTMX pattern: poslední řádek má hx-get="...?page=N+1" hx-trigger="revealed" hx-swap="afterend". Nízká
Tests (Vitest + Testing Library) Spring MockMvc + @WebMvcTest + HtmlUnit / Selenide / Playwright pro E2E. Žádné JS unit testy (minimum klient. logiky). Backendové testy se rozšíří o rendering. Střední

5. Cílová architektura (dual-UI)

Doporučená cílová architektura ponechává API pro 3rd party a přidává HTMX UI jako druhý primary adapter:

backend/
├── *.api/                  ← současný HAL+FORMS REST (zachováno pro 3rd party)
│   └── /api/v1/members     ← JSON, bearer token auth
│
├── *.web/                  ← NOVÝ Thymeleaf + HTMX UI
│   ├── /members            ← HTML, session cookie auth
│   ├── /members/{id}/edit  ← HTMX fragmenty
│   └── templates/
│       ├── layout/main.html
│       ├── members/list.html
│       └── members/detail.html
│
└── common/security/
    ├── oauth2-resource-server   ← pro /api/**
    └── form-login + session     ← pro /**

Obě UI sdílí doménové příkazy (MemberCommandService, EventCommandService). Rozdíl je jen v adapteru: REST controller vs. MVC controller. Díky hexagonální architektuře, kterou projekt má, je to přímočaré.

5.1 Dva bezpečnostní filter-chainy

@Bean
@Order(1)
SecurityFilterChain apiChain(HttpSecurity http) {
    http.securityMatcher("/api/**")
        .oauth2ResourceServer(o -> o.jwt(...))
        .csrf(c -> c.disable());
    return http.build();
}

@Bean
@Order(2)
SecurityFilterChain webChain(HttpSecurity http) {
    http.securityMatcher("/**")
        .formLogin(f -> f.loginPage("/login"))
        .oauth2Login(Customizer.withDefaults())   // volitelně, pokud chceme OIDC i pro UI
        .csrf(Customizer.withDefaults());         // CSRF zapnuto
    return http.build();
}

6. OAuth2/OIDC migrace — nejtvrdší oříšek

Klíčové rozhodnutí. HTMX posílá requesty jako HTML formuláře — hlavičku Authorization: Bearer do nich dostanete jen přes JavaScript (htmx:configRequest). Oficiální HTMX doporučení: „nepoužívejte bearer tokeny v localStorage pro UI, použijte secure HttpOnly cookie“. To ale znamená, že UI už není OAuth2 client SPA, ale klasická session-based aplikace.

6.1 Tři možné přístupy

VariantaPopisProsCons
A. Session-only UI Form login na /login, HttpOnly session cookie. API endpointy ponechány jako OAuth2 resource server (pro 3rd party). Nejjednodušší. CSRF out-of-the-box. Žádný JS token management. Plně Spring Security standard. Dvě cesty autentizace (session pro UI, bearer pro API). Uživatel musí mít samostatný flow pro 3rd party.
B. OAuth2 Client + session UI je Spring oauth2Login() — Spring sám provede authorization code flow se svým vlastním auth serverem, access token si drží v HttpSession, cookie je jen session ID. Uživatel vidí stejnou login obrazovku. Přihlášení přes OIDC (budoucí sociální login). Auth server zůstává autoritou. Trochu overkill — auth server a UI běží ve stejném procesu. Nutno opatrně s registered client pro „sebe samu“.
C. Token v cookie + bearer hybrid Auth server vystaví JWT, backend ho uloží do HttpOnly cookie. Filter ho při každém requestu zkopíruje do Authorization hlavičky. Jeden auth mechanismus pro UI i API. Konzistentní. Vlastní kód pro token refresh. Riziko CSRF (nutno double-submit cookie). Netriviální.
Doporučení: varianta B (OAuth2 Client + session). Uživatel se přihlásí stejným flow jako dnes (Spring AS na /oauth2/authorize), ale v redirect URI místo SPA callbacku Spring sám dokončí flow, uloží OAuth2AuthorizedClient a vytvoří HTTP session. Výhoda: pokud se rozhodnete přidat další OIDC providery (Google, GitHub), stačí je přidat jako ClientRegistration.

6.2 Dopad na existující bezpečnostní rozšíření

  • KlabisAuthorizationServerCustomizer — zůstává beze změny (AS je autonomní modul).
  • JWT claims (registrationNumber, memberIdUuid, authorities) — pro UI se načítají z OAuth2AuthorizedClient při loginu, uloží se do Authentication.principal (stejné Member resolvní přes CurrentUserArgumentResolver).
  • Field-level authorization (@OwnerVisible, @HasAuthority, PatchField) — funguje beze změny, pracuje s Authentication, ne s requestem.
  • local-dev profil (refresh-token workaround) — odpadá! Session-based UI žádný refresh token nepotřebuje.
  • TokenRenewalOverlay — odpadá. Session prodlužování je triviální (request prodlužuje session).

7. HAL+FORMS vs. server-rendered UI

HAL+FORMS je silná abstrakce, protože frontend dynamicky renderuje formuláře podle _templates. Po migraci na HTMX tato abstrakce už není nutná pro UI — server má plnou kontrolu nad HTML. Ale HAL+FORMS zůstává pro 3rd party API.

Co z HAL+FORMS v UI zanikne

  • Dynamický render formuláře z JSON template.
  • HalFormsFieldFactory — nahrazeno Thymeleaf fragmenty per-field.
  • enrichTemplateWithReadOnlyFields — není potřeba, server zná kontext.
  • Klient. navigace přes _links — nahrazeno normálními <a> a hx-get.

Co zůstane

  • HAL+FORMS API endpointy (Accept: application/prs.hal-forms+json) pro 3rd party.
  • Všechny klabisLinkTo a klabisAfford v *.api controllerech.
  • Field-level authorization (PatchField, @OwnerVisible) — pracuje na úrovni záznamu, funguje stejně v obou vrstvách.
Pozor na duplicitu. Pokud UI nepoužije HAL+FORMS, bude nutné každý formulář definovat dvakrát: jednou jako Thymeleaf template pro UI, podruhé jako HAL+FORMS _templates pro API. To je cena za udržení obou protokolů. Mitigace: generovat HAL+FORMS _templates z Bean Validation anotací na command objektech (už se tak děje pro UI formuláře).

8. Možné migrační cesty

Cesta 1 — Big Bang (NEDOPORUČENO)

Kompletní přepis za jednu iteraci. Aplikace je v rané fázi, ale má už 15+ stránek a netriviální security. Odhad 8–12 týdnů, během kterých nelze dodávat nové funkce.

  • Risk vysoký
  • Výhoda: čistý kód, bez duálního deploymentu

Cesta 2 — Paralelní UI, postupná migrace (DOPORUČENO)

  1. Fáze 0 (1 týden): Přidat Thymeleaf + htmx-spring-boot dependency. Vytvořit web package vedle api. Form login SecurityFilterChain pro /ui/**.
  2. Fáze 1 (1 týden): Sdílený layout: nav, footer, toast, modal container. hx-boost na body. Theme toggle v Alpine.js.
  3. Fáze 2 (2 týdny): Migrace read-only stránek: Members list, Events list, Calendar, Dashboard. Zatím bez editace.
  4. Fáze 3 (3 týdny): Members detail + edit (click-to-edit), Member registration (multi-step). Řeší složitější formulářový UX.
  5. Fáze 4 (2 týdny): Groups, Training Groups, Family Groups (stejný vzor jako Members).
  6. Fáze 5 (1 týden): Finances, Category Presets.
  7. Fáze 6 (1 týden): Přepnutí default URL z React SPA na HTMX UI. React SPA zůstává jako /legacy nebo odstraněn.
  8. Fáze 7 (0.5 týdne): Smazání frontend/, static/, build scriptů. Aktualizace CLAUDE.md a developer manuálu.

Odhad: 10–12 týdnů s jedním vývojářem. Během migrace lze dodávat nové funkce do obou UI (ale nové features dělat rovnou v HTMX).

Cesta 3 — „HTMX island“ v React SPA

Ponechat React shell a postupně nahrazovat jednotlivé stránky server-rendered HTML loadovaným přes HTMX. NEDOPORUČENO — ztrácí výhody obou přístupů, složitý security model (bearer i session současně).

9. Rizika a kompromisy

OblastRizikoMitigace
OAuth2 session architektura Špatně nastavený oauth2Login může rozbít 3rd party API auth. Dva SecurityFilterChain s securityMatcher. Kompletní integrační testy pro oba flow-y.
Komplexní formulářový UX Multi-step wizardy, conditional fields, cross-field validation — v Reactu trivialita, v HTMX víc server roundtripů. Pro opravdu složité wizardy (např. member registration) použít Alpine.js x-data s klientským stavem, HTMX jen pro finální submit.
Calendar UX HTMX není ideální pro drag-drop, inline editaci událostí, real-time update. Přijmout, že calendar zůstane klient-heavy — použít FullCalendar JS. HTMX pro CRUD operace.
Duplicita formulářů HAL+FORMS v API + Thymeleaf v UI = dvojí údržba. Generovat HAL+FORMS _templates z Bean Validation (Spring HATEOAS to umí); sdílet command objekty.
Offline-first / PWA Pokud se bude chtít offline režim (např. terénní sběr dat na závodech), HTMX tomu nepomáhá. Projekt to aktuálně nepotřebuje. Pokud se objeví, je řešením samostatná mini-SPA pro offline use-case, ne návrat k Reactu.
Testování Ztrácí se Vitest + Testing Library. MockMvc testuje HTML fragmenty textově (brittle). Pro kritické UI flow-y psát Playwright E2E testy (už máme QA skill). Unit testy přesunout na controllery a command handlery.
Team skill Tým je zvyklý na React; HTMX + Thymeleaf je jiná mentální mapa (hypermedia-first). Pilot na jedné stránce (např. Dashboard) před commit k plné migraci.
3rd party JS knihovny Date pickers (formik-mui-x-date-pickers), rich editor, atd. Nativní <input type="date"> je dostatečný pro 90% případů; pro zbytek Flatpickr/Tom-Select (vanilla).
Ztráta typesafety mezi BE a FE OpenAPI generator dnes produkuje TypeScript typy. Není aplikovatelné — FE je v Javě. Templaty mají přístup k doménovým typům nativně.

10. Doporučení

Migrace je technicky proveditelná a dává smysl, pokud primární motivací je snížit komplexitu a zrychlit vývoj featur. Spring Boot má vyzrálý ekosystém pro HTMX (htmx-spring-boot), hexagonální architektura projektu umožňuje přidat druhý UI adapter bez dotyku domény.

Go/no-go kritéria

Jděte do migrace, pokud:

  • Tým preferuje full-stack Java nad TypeScriptem.
  • Plánované UI je primárně CRUD + tabulky + formuláře (to odpovídá Klabis).
  • Není plán na mobilní aplikaci sdílející frontend kód.
  • Není vyžadována real-time collaboration (multi-user editace).

Nejděte do migrace, pokud:

  • Plánuje se progresivně vyměnit za mobilní app (React Native by sdílel komponenty) — u Klabis zřejmě neplatí.
  • UI se posune směrem k bohatým klientským interakcím (kreslení na závodní mapy, drag-drop trasy).

Navrhovaný další krok

  1. Spike (1 týden): Pilot na jedné stránce — doporučuji Members list + detail (read-only). Ověří Thymeleaf render, hx-boost, session auth, layout sdílení.
  2. Rozhodovací bod: po spike vyhodnotit produktivitu (řádky kódu, rychlost vývoje, UX dojem).
  3. Pokud pozitivní → Cesta 2 (paralelní migrace) podle fází výše.
  4. OpenSpec proposal pro každou fázi, aby se zachovala auditní stopa.
