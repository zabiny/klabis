Architektura projektu

Klabis je modulární monolit postavený nad Spring Modulith. Každý modul má vlastní veřejné API (kořenový package) a tři interní package domain / application / infrastructure dle hexagonální architektury. Pravidla závislostí mezi moduly vynucuje Spring Modulith.

Aplikační moduly

Moduly jsou top-level package pod com.klabis. Každý je označen @ApplicationModule v package-info.java: 

@org.springframework.modulith.ApplicationModule
package com.klabis.members;
ModulÚčel
commonSdílené jádro (security, hateoas, persistence helpery, users, …) — používá ho každý modul.
membersČlenové, registrace, OAuth2 server customizace, ActingMember.
eventsZávody a další "události" v kalendáři.
calendarSjednocený pohled na kalendář (události + manuální položky).
groupsSkupiny členů.
orisAdaptér pro externí ORIS API (orientační běh).
Tato dokumentace popisuje pouze common + framework-like části members (ActingMember, OAuth2 customizace). Byznys logiku ostatních modulů hledejte ve specifikacích v openspec/specs/.

Hexagonální struktura modulu

Každý modul je rozdělen do tří package:

{module}/ (root)
Veřejné API modulu — typy a interfaces, které smí používat jiné moduly: hodnotové typy (např. MemberId), DTO pro publikované eventy (MemberCreatedEvent), meta-anotace (@ActingMember).
Modulith považuje vše mimo root (kromě explicitně exportovaných package) za interní.
{module}/domain/
Doménový model: agregáty, value objects, repository interfaces, domain events. Bez závislostí na Spring frameworku (smí jen org.springframework.data.domain a jMolecules).
{module}/application/
Application services (use cases), @PrimaryPort interfaces a jejich implementace, konfigurace modulu, cross-aggregate koordinace, transakční hranice.
{module}/infrastructure/
Adaptéry — REST controllery (restapi/), JDBC repositories (jdbc/), event listeners (listeners/), bootstrap (bootstrap/), externí integrace.

jMolecules anotace

Pro vyznačení role typu používáme jMolecules. Anotace jsou jen metadata (žádný runtime efekt), ale poskytují čitelnost a podklad pro statickou analýzu.

@org.jmolecules.architecture.hexagonal.PrimaryPort
Interface application service volaný "zvenku" (REST, scheduled job, jiný modul).
@org.jmolecules.architecture.hexagonal.SecondaryPort
Interface, který doména/application volá ven (typicky repository, externí gateway).
@org.jmolecules.architecture.hexagonal.PrimaryAdapter
Implementace primárního adaptéru — typicky REST controller.
@org.jmolecules.architecture.hexagonal.SecondaryAdapter
Implementace sekundárního adaptéru — typicky JDBC repository.
@org.jmolecules.ddd.annotation.AggregateRoot, @ValueObject, @Repository, @DomainEvent
DDD building blocks (přímo na třídách v domain/).

Příklad portu:

@org.jmolecules.architecture.hexagonal.PrimaryPort
public interface RegistrationPort {
    record RegisterNewMember(...) {}

    @Transactional
    Member registerMember(RegisterNewMember command);
}

Pravidla závislostí

  • Žádný modul nesmí importovat infrastructure/application jiného modulu. Pouze veřejné API (kořenový package) cílového modulu.
  • domain nesmí záviset na infrastructure ani na Spring frameworku (kromě spring-data-commons a jMolecules).
  • application může záviset na domain a na veřejném API jiných modulů.
  • infrastructure má největší volnost — implementuje porty z application a domain a vidí na všechny Spring/JDBC/Jackson knihovny.
  • Cross-modul integrace probíhá pouze přes domain eventy (viz kap. 08).
Pravidla automaticky ověřuje Spring Modulith. Pokud chcete strukturu zkontrolovat, spusťte ApplicationModulesTest v testech (ten obsahuje ApplicationModules.of(KlabisApplication.class).verify()).
