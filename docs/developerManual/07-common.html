Common — sdílené stavební bloky

Tato kapitola popisuje znovu-použitelné domain/infrastructure bloky, které sdílí všechny moduly: UserGroup (kompozice pro skupiny s členstvím a invitation flow), EmailService a TemplateRenderer pro odesílání emailů a doménovou hierarchii výjimek mapovanou na HTTP statusy.

UserGroup building block

common.usergroup obsahuje znovu-použitelný stavební blok pro skupiny uživatelů s vlastnictvím (owners) a členstvím (members). Není to agregát — identita patří agregátu, který UserGroup embeduje (kompozice). Aktuálně ho používají agregáty FamilyGroup, TrainingGroup a FreeGroup v modulu groups.

UserGroup

Drží jméno skupiny, množinu vlastníků (UserId) a množinu členství (GroupMembership). Bez závislosti na Spring frameworku — čistě doménový kód. 

public class UserGroup {
    public static UserGroup create(String name, UserId owner);
    public static UserGroup reconstruct(String name, Set<UserId> owners, Set<GroupMembership> members);

    public void rename(String newName);
    public void addMember(UserId userId);
    public void removeMember(UserId userId);
    public void addOwner(UserId userId);
    public void removeOwner(UserId userId);

    public boolean isOwner(UserId userId);
    public boolean hasMember(UserId userId);
    public boolean isLastOwner(UserId userId);

    public String getName();
    public Set<UserId> getOwners();
    public Set<GroupMembership> getMembers();
}

Použití přes kompozici

public class FamilyGroup extends KlabisAggregateRoot<FamilyGroup, FamilyGroupId> {
    private final FamilyGroupId id;
    private final UserGroup userGroup;

    public static FamilyGroup create(CreateFamilyGroup command) {
        UserGroup userGroup = UserGroup.create(command.name(), command.ownerId());
        return new FamilyGroup(FamilyGroupId.newId(), userGroup);
    }

    public void addMember(UserId userId) {
        userGroup.addMember(userId);
        registerEvent(...);
    }
}
Výhoda kompozice (vs. dědičnost): různé agregáty mají různé další chování (FamilyGroup automaticky přidá ownera i jako membera, TrainingGroup ne) a nepotřebují dědit konkrétní invariants.

Invariants vynucované UserGroup

  • Jméno skupiny nesmí být prázdné.
  • Skupina musí mít aspoň jednoho vlastníka. Pokus o odstranění posledního vyhodí CannotRemoveLastOwnerException.
  • Vlastník nemůže být odstraněn jako člen (pouze jako vlastník) — OwnerCannotBeRemovedFromGroupException.
  • Stejný uživatel nemůže být přidán dvakrát — MemberAlreadyInGroupException.
  • Odstranění neexistujícího člena vyhodí MemberNotInGroupException.

WithInvitations — invitation flow

Některé skupiny pracují s pozvánkami místo přímého přidání člena. Pro ně existuje samostatný contract WithInvitations. Společné výjimky: DirectMemberAdditionNotAllowedException, NotInvitedMemberException, DuplicatePendingInvitationException, InvitationNotFoundException, CannotInviteExistingMemberException. 

public interface WithInvitations {
    void invite(UserId invitedBy, UserId target);
    void acceptInvitation(InvitationId invitationId);
    void rejectInvitation(InvitationId invitationId);

    List<Invitation> getPendingInvitations();
    Set<Invitation> getInvitations();
    boolean isInvitedMember(UserId userId);
    boolean hasMember(UserId userId);
}

Invitation entity

@Entity
public class Invitation {
    public static Invitation createPending(UserId invitedBy, UserId invitedUser);

    public void accept();          // PENDING → ACCEPTED
    public void reject();          // PENDING → REJECTED
    public boolean isPending();
    public boolean isForUser(UserId userId);

    // gettery: id, invitedUser, invitedBy, status, createdAt
}

InvitationStatus = PENDING | ACCEPTED | REJECTED. Po opuštění PENDING stavu je další transition zakázána (vyhodí IllegalStateException).

GroupMembership

Lehký value object popisující členství uživatele ve skupině (UserId + okamžik vstupu). Slouží pro rozlišení vlastníka (Set<UserId>) od běžných členů (Set<GroupMembership>).

Persistence sdíleného UserGroup

Aktuálně mají všechny tři group agregáty (FamilyGroup, TrainingGroup, FreeGroup) sdílenou JDBC reprezentaci (společný GroupMemento + GroupJdbcRepository v modulu groups) s type discriminator sloupcem. Každý agregát má vlastní repository adapter, ale fyzicky leží data v jedné tabulce.

EmailService

public interface EmailService {

}
JavaMailEmailService
Reálné odesílání přes SMTP. Aktivní pouze s profilem email a nastavenými KLABIS_SMTP_* env vars.
LoggingEmailService
Výchozí v dev/test — email se vypíše do logu místo odeslání. Užitečné pro vývoj bez SMTP serveru.
Implementace nikdy nesmí vyhodit výjimku — neúspěch se loguje. Email selhání nesmí blokovat byznys operaci. Pro dev s reálnými emaily lze použít MailHog (docker compose up mailhog -d).

Hierarchie doménových výjimek

Výjimky v common.exceptions jsou základní, doménové moduly z nich dědí. Mapování na HTTP status řeší MvcExceptionHandler (kap. 05).

BusinessRuleViolationException (abstract)
Base pro porušení byznys pravidel — HTTP 400. Doménové moduly z ní dědí (např. InvalidUpdateException, MemberAlreadyInGroupException).
InvalidDataException
Nevalidní vstupní data (jiné než Bean Validation chyby) — HTTP 400.
AuthorizationException
Obecné selhání autorizace — HTTP 403.
InsufficientAuthorityException
Uživatel nemá konkrétní autoritu — HTTP 403.
ResourceNotFoundException
Zdroj neexistuje — HTTP 404. Doménové moduly z ní dědí (MemberNotFoundException, …).
MemberProfileRequiredException
Akce vyžaduje Member profil, ale uživatel ho nemá — HTTP 403. Vyhazuje resolver @ActingMember.
