Testování

Testovací stack stojí na JUnit 5 + AssertJ + Mockito + Spring Boot test slicy. Klabis přidává pár vlastních anotací a assertion tříd, které řeší specifické potřeby projektu — autentizaci v testech, izolaci E2E flow a doménové asserty.

Pyramida v Klabisu

VrstvaAnotaceCo testuje
Doménový unitplain JUnit + MockitoAgregáty, value objects, domain services. Bez Springu.
Repository@DataJdbcTestJDBC repository adaptery + memento mapování.
Controller@WebMvcTest + @WithKlabisMockUserEndpoint mapping, request/response, autorizace.
Modulová integrace@ApplicationModuleTestModul + jeho závislosti, event flows uvnitř modulu.
End-to-end@E2ETestPlný flow přes všechny moduly s MockMvc.
Modularitaplain test s ApplicationModulesVerifikace pravidel závislostí.
Detailní popis pyramidy a kdy co volit najdete v docs/testing-pyramid.md. Tato kapitola pokrývá pouze Klabis-specific nástroje.

@WithKlabisMockUser

Anotace pro autentizaci v testech (analog Spring @WithMockUser, ale produkuje KlabisJwtAuthenticationToken s typovaným UserId, memberIdUuid a Authority). Vždy preferujte tuto anotaci před @WithMockUser — ta vytváří plain token bez Klabis claims a field-level autorizace ji nepozná. 

@Retention(RUNTIME)
@Target({METHOD, TYPE})
@WithSecurityContext(factory = WithKlabisMockUserSecurityContextFactory.class)
public @interface WithKlabisMockUser {
    String userId()    default "";   // pokud chybí, vygeneruje se random
    String memberId()  default "";   // pokud chybí, uživatel nemá Member profil
    String username()  default "";   // pro Authentication.getName()
    Authority[] authorities() default {};
}
@Test
@WithKlabisMockUser(authorities = {Authority.MEMBERS_MANAGE})
void admin_can_update_any_member() { ... }

@Test
@WithKlabisMockUser(memberId = OWNER_ID_STRING)
void owner_can_see_own_birth_number() { ... }
Pokud má memberId hodnotu a userId chybí, použije se stejná hodnota i pro userId (member ID = user ID podle shared-ID konvence — viz kap. 02). Pokud testujete ownership-based field security, musíte nastavit memberId.

@E2ETest

Meta-anotace pro end-to-end testy. Spustí celou aplikaci se všemi moduly a poskytne MockMvc pro HTTP volání. 

@ApplicationModuleTest(mode = ALL_DEPENDENCIES, verifyAutomatically = false)
@AutoConfigureMockMvc
@ActiveProfiles("test")
@CleanupTestData
@Import(TestApplicationConfiguration.class)
@Retention(RUNTIME)
@Target(TYPE)
public @interface E2ETest {}
BootstrapMode.ALL_DEPENDENCIES
Načte tested modul + všechny moduly, na kterých závisí (přechodně). Listenery v jiných modulech budou reagovat na publikované eventy.
verifyAutomatically = false
Vypne kontrolu závislostí mezi moduly. Tu má dělat samostatný ApplicationModulesTest, ne každý E2E test.
@CleanupTestData
Listener, který po každém testu vymaže data z DB (E2E testy commitují kvůli event flow, proto je nutný explicitní cleanup).
@E2ETest
class MemberRegistrationE2ETest {
    @Autowired MockMvc mvc;

    @Test
    @WithKlabisMockUser(authorities = {Authority.MEMBERS_MANAGE})
    void registration_creates_user_and_sends_password_email() throws Exception {
        mvc.perform(post("/api/registration").contentType(...).content(json))
            .andExpect(status().isCreated());
        // … asserty na DB i emailovou frontu …
    }
}

@ApplicationModuleTest pro modul

Použijte pro testy uvnitř modulu, kdy chcete reálné Spring beany ale s izolací od ostatních modulů. Pro listenery se hodí PublishedEvents a Scenario ze Spring Modulith. 

@ApplicationModuleTest
class CalendarUpdateOnEventChangeTest {
    @Test
    void event_change_creates_calendar_item(Scenario scenario) {
        scenario.publish(new EventCreatedEvent(...))
                .andWaitForStateChange(() -> calendarRepo.findByEventId(...))
                .andVerify(item -> assertThat(item).isPresent());
    }
}

Custom assertions (AssertJ)

Pro doménové typy existují AbstractAssert rozšíření, která dělají testy čitelnější. Aktuálně: MemberAssert, UserAssert, EventAssert, CalendarItemAssert. 

import static com.klabis.members.MemberAssert.assertThat;

assertThat(savedMember)
    .hasFirstName("Karel")
    .hasLastName("Vomáčka")
    .hasRegistrationNumber("ZBM2501");

Pro nový agregát napište vlastní {Aggregate}Assert v src/test/java (stejný package jako agregát). Pokrývá nejčastější asserty + skrývá detaily getterů.

Memento round-trip test

Pro každý agregát mít test, který ověří, že memento přesně zrcadlí doménový stav (po from(aggregate).toAggregate() musí vyjít equální agregát). Chrání před chybami při přidávání nových polí. 

@Test
void memento_roundtrip_preserves_state() {
    Member original = MemberTestDataBuilder.aMember().withFirstName("Karel").build();
    Member roundtripped = MemberMemento.from(original).toMember();

    assertThat(roundtripped).isEqualTo(original);   // identita podle ID
    MemberAssert.assertThat(roundtripped).hasFirstName("Karel");
}

Test data buildery

Pro každý agregát existuje fluent builder ({Aggregate}TestDataBuilder) v testovacím classpath, který poskytuje sensible defaults a umožňuje nastavit jen relevantní pole testu. Eliminuje opakující se setup kód.

Spouštění testů

Vždy používejte agenta test-runner (skill developer:test-runner-skill) místo přímého ./gradlew test. Důvody: zabraňuje paralelnímu Gradle (deadlock na cache lock), filtruje výstup, předává jen relevantní failures.

Gotcha: @WebMvcTest a UserService

SecurityConfiguration.jwtAuthenticationConverter() potřebuje UserService bean. V @WebMvcTest kontextu přidejte: 

@WebMvcTest(MemberController.class)
class MemberControllerTest {
    @MockitoBean UserService userService;
    @MockitoBean OwnershipResolver ownershipResolver;   // pokud testujete @OwnerVisible
    // …
}
