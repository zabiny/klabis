Práce s Claude Code

Klabis je do velké míry vyvíjen s asistencí Claude Code. Projekt obsahuje sadu OpenSpec specifikací, skills, agentů a commands, které celý workflow udržují konzistentní. Tahle stránka popisuje typické postupy: jak přidat feature, jak refaktorovat, kdy spouštět které agenty.

Filozofie projektu

Workflow staví na třech pilířích:

  1. Specifikace první — než se píše kód, popíše se chování v openspec/specs/. Změny se navrhují přes change proposals, ne přímou editací specs.
  2. Konvence v skillech — opakující se patterny (agregát, controller, JDBC adapter) jsou zdokumentované v backend-patterns skillu, který si Claude automaticky načítá.
  3. TDD a malé iterace — feature se rozdělí na vertikální slice (jeden flow přes všechny vrstvy), každý slice má své testy, je samostatně commitovatelný.

OpenSpec workflow

OpenSpec je systém pro řízení specifikací a změnových návrhů. V Klabisu žije v openspec/:

openspec/specs/
Aktuální specifikace aplikace, organizované po bounded contextech (modulech). Každý modul má jeden spec.md psaný z user-facing perspektivy (scénáře, ne implementace).
openspec/changes/
Otevřené change proposals (ještě nezaarchivované). Každý je samostatný adresář s proposal.md, design.md, tasks.md a delta speciacemi.
openspec/changes/archive/
Hotové změny — slouží jako historie a kontext pro Claude.
openspec/config.yaml
Pravidla pro psaní spec (jazyk, scope, styl scénářů).

Typický cyklus změny

  1. /openspec-explore nebo /openspec-new-change — nahodit nápad, prozkoumat kódovou základnu, ujasnit si požadavky.
  2. /openspec-propose — vytvoří proposal + design + tasks v jednom kroku (nebo postupně přes /openspec-continue-change).
  3. /openspec-ff-change — fast-forward přes všechny artefakty (rychlý mód, pokud je proposal už promyšlený).
  4. /openspec-apply-change nebo /openspec-ext:implement-proposal — implementace podle tasks.md. implement-proposal používá víc subagentů iterativně.
  5. /openspec-verify-change — zkontrolovat, že implementace odpovídá specifikaci.
  6. /openspec-archive-change — zaarchivovat hotovou změnu (přesun do archive/) a sloučit delta do hlavních specs.
Pro malé bug fixy nebo čistě technické refactoring není nutné zakládat OpenSpec change. Použijte tasks/ queue (níže) nebo přímou implementaci přes backend-developer agenta.

Agenti

Agenti jsou prekonfigurované role s vlastními skills a chováním. V projektu (.claude/agents/):

backend-developer
Java vývojář pro Spring/Modulith. Auto-load: backend-patterns, developer:spring-conventions, developer:tdd-best-practices. On-demand: spring-modulith, spring-data-jdbc, spring-hateoas-api, ddd-domain-modeling, … Použijte na jakoukoli implementaci/oprava backend kódu.
frontend-developer
React/TypeScript vývojář. Auto-load: hal-navigator-patterns, developer:test-runner-skill, developer:tdd-best-practices. Pre-flight: vždy nejdřív projde existující patterny.
git-operator
Git operace (commits, branching, rebase). Použijte na cokoli git-related, abyste se vyhnuli chybám s 1Password signing apod.
Externí marketplace agenti, které se hodí: Explore (rychlé prohledání kódu), Plan (architektonický plán), developer:test-runner, developer:code-reviewer.

Slash commands

V .claude/commands/:

/frontend-qa-testing
Plánuje a spouští QA testy podle openspec specifikací (přes Playwright).
/simplifyModule
Hluboká analýza modulu — duplikace, dead code, over-abstrakce. Užitečné po větší feature nebo před refactoringem.
/tests-structure-review
Review struktury testů (pyramida, smoke testy).

Jak přidat novou feature (typický flow)

  1. Spec: /openspec-new-change nebo /openspec-propose — popsat scénáře z pohledu uživatele, design rozhodnutí, rozdělit na tasks.
  2. Implementace: spustit backend-developer agenta s odkazem na tasks.md. Pro frontend totéž s frontend-developer.
  3. Vertikální slice: jedna feature = jeden end-to-end flow napříč vrstvami. Po každé fázi (commit-able chunk) commit a test run.
  4. Testy: TDD cyklus (test první). Vždy přes developer:test-runner-skill.
  5. Code review: před commitem spustit code-review skill (nebo developer:code-reviewer agenta).
  6. QA: /openspec-ext:qa-testing nebo developer:frontend-qa-testing přes Playwright.
  7. Verify + archive: /openspec-verify-change → /openspec-archive-change.

Refactoring

  1. Pokud refactoring mění chování, založit OpenSpec change. Pokud je čistě technický (jiná struktura, žádná změna chování), stačí task queue.
  2. Vertikální slice (backend/CLAUDE.md): refaktorovat jednu feature/metodu end-to-end přes všechny vrstvy, ne jednu vrstvu napříč všemi feature.
  3. Po každém slice testovat (test-runner) a commitnout.
  4. Po dokončení modulu spustit /simplifyModule a/nebo domain-review-to-task pro audit.

Lokální spouštění

./runLocalEnvironment.sh
# spustí backend (https://localhost:8443) a frontend (http://localhost:3000)
Pro testování UI vždy port 3000 (Vite HMR auto-reload). Port 8443 obsahuje statický bundle, který se aktualizuje až po publish-frontend-resources + restartu backendu.

Testovací uživatelé

  • Admin: registration ZBM9000, heslo password.
  • Member: registration ZBM9500, heslo password.

Sandbox a ostatní pravidla

  • Nikdy nepoužívejte dangerouslyDisableSandbox. Pokud command v sandboxu selže, reportněte chybu — uživatel rozhodne.
  • Při paralelizaci: nikdy nespouštějte víc Gradle/test-runner agentů paralelně (deadlock na cache lock).
  • Git commits: nikdy --no-verify, nikdy --no-gpg-sign (1Password signing musí projít).
  • Pro JetBrains MCP tools: nepoužívat v git worktree (selžou).

Ramdisk pro temp soubory

Pro dočasné soubory používejte /mnt/ramdisk/klabis/ — rychlejší než disk a uklízí se samo při restartu.
