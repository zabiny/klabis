Spring Modulith a moduly

Komunikace mezi moduly probíhá výhradně přes domain eventy. Spring Modulith publikuje eventy přes transactional outbox pattern (tabulka event_publication). Listenery v jiných modulech reagují asynchronně, mají vlastní transakci a automatický retry. Tato kapitola pokrývá způsob, jakým moduly Klabisu komunikují, publikaci, listenery, idempotenci a observability metriky.

Publikace eventu

Event se registruje na agregátu pomocí registerEvent(...). Spring po úspěšném save() a commitu transakce vytvoří záznamy v event_publication a předá eventy listenerům. 

public class Member extends KlabisAggregateRoot<Member, MemberId> {
    public static Member registerNewMember(...) {
        Member m = new Member(...);
        m.registerEvent(MemberCreatedEvent.fromAggregate(m));
        return m;
    }
}
Event třída leží v kořenovém package modulu (je veřejné API), nikoli v domain/ — aby ji mohly importovat jiné moduly.

@ApplicationModuleListener

Listener je @Component v {module}/infrastructure/listeners/. @ApplicationModuleListener automaticky:

  • Spustí handler v nové transakci (REQUIRES_NEW), takže selhání nebrání commitu publikujícího agregátu.
  • Zařídí asynchronní doručení.
  • Při výjimce zaznamená neúspěch v outboxu a Spring Modulith ho po restartu znovu publikuje (incomplete event detection).
@Component
@PrimaryAdapter
public class UserCreatedEventHandler {

    @ApplicationModuleListener
    public void onUserCreated(UserCreatedEvent event) {
        if (event.accountStatus() != AccountStatus.PENDING_ACTIVATION) return;
        if (event.email().isEmpty()) return;
        // … vygeneruj token, odešli email …
    }
}

Transactional outbox

Spring Modulith ukládá publikované eventy do tabulky event_publication (V003 migrace). Klabis používá tyto klíčové vlastnosti:

  • completion-mode: UPDATE — po úspěchu se publikace označí jako kompletní, ne smazána (= audit trail).
  • republish on restart — incomplete eventy starší než 5 minut se po startu aplikace znovu publikují.
  • cleanup po 7 dnech — kompletní záznamy se mažou.
Konfigurace je v application.yml pod spring.modulith.events.*. Detailní popis flow viz backend/docs/EVENT-DRIVEN-ARCHITECTURE.md.

Idempotence

Listener musí být idempotentní — Spring Modulith může událost doručit víckrát (po výjimce, restartu). Typické vzory:

  • Kontrola "už existuje?" před vytvořením (např. pomocí stabilního ID z eventu).
  • Stavový check (jako v ukázce výše: ignorovat, pokud uživatel už není PENDING_ACTIVATION).
  • Idempotentní operace (např. přidat člena do skupiny — pokud už tam je, nevyhodí výjimku).

Konvence event flow

  1. Doménová akce → agregát zaregistruje event přes registerEvent(...).
  2. save(aggregate) v application service spustí publikaci.
  3. Spring Modulith vloží záznam do outboxu, commit transakce.
  4. Po commitu se eventy doručí všem @ApplicationModuleListener.
  5. Listener si načte vlastní data přes svůj application service (ne z eventu — event nese pouze ID).
  6. Pokud listener selže, outbox záznam zůstane "incomplete" a Modulith ho po restartu zopakuje.
Eventy nemají obsahovat byznys data, jen ID a minimální payload nutný k rozhodnutí (např. typ akce). Pokud listener potřebuje detail, načte si ho přes service. Eliminuje se tím riziko zastaralého snapshotu v eventu.

Observability

Profil metrics aktivuje CustomMetricsConfiguration, který expose Micrometer metriky:

MetrikaVýznam
spring.modulith.events.publishedCounter — celkový počet publikovaných eventů.
spring.modulith.events.incompleteGauge — aktuální počet nedokončených eventů (backlog).
spring.modulith.events.latencyHistogram doby zpracování (ms).

Dostupné na /actuator/prometheus. CustomMetricsTrackingAspect obaluje listenery a měří jejich latence.

Verifikace pravidel závislostí

Pravidla závislostí mezi moduly (kap. 01) automaticky ověřuje Spring Modulith. V projektu existuje ApplicationModulesTest, který obsahuje: 

ApplicationModules.of(KlabisApplication.class).verify();

Tento test selže, pokud modul importuje nepublikovaný typ z jiného modulu (např. events.application z members.application).
